## 二、逐项回复

### 🔴 P0：招生专员全部 403 — 确认真实 bug

**测试正确。** 权限矩阵中 recruiter 拥有完整权限（leads、courses、enrollments、recommendations、workers 全部在 allow 列表），但所有 API 返回 403。

**根因**：`verifyToken` 环节。招生专员登录后 token 有效，但用 anon key 查询 Supabase `users` 表获取 role 时失败，导致 token 校验降级为 `guest`，`guest` 不在任何权限矩阵中 → 全部 403。

这是一个 token 验证层的 bug，不是权限设计问题。

**修复计划**：P0，立即修复。

---

### ⚠️ 讲师课程评价 404/405 — 测试端点和业务设计不一致

测试请求了 `GET/POST /api/courses/reviews`，但根据业务逻辑全景图：

### ⚠️ 客户无法查看阿姨列表（403）— 这是正确业务设计

**不是 bug。**

业务逻辑全景图 §2 明确：
> 客户核心定位：被服务对象，**只看已签约订单**
> 关键权限：`orders:read`（仅自己的）

§5 数据隔离矩阵中，客户对"阿姨库"的权限为 **"—"（空，无权限）**。

客户只能通过已签约订单查看已签约阿姨的详情（`/api/workers/{id}` 是公开端点），**不应浏览阿姨自由市场列表**。这是平台设计的隔离策略，防止客户绕过经纪人直接联系阿姨。

**结论：设计行为，建议从测试清单中移除此项。**

## 三、为什么自动化 751 条没测出来？

三个原因：

**1. 自动化跳过了 token 验证链路**
- 自动化测试使用硬编码 `dev-admin` token，完全绕过 `verifyToken` 的 Supabase 用户查询
- 招生专员 403 的根因在 token 验证失败/降级，自动化测试根本不会触发这个代码路径

**2. 自动化仅覆盖"已注册 API 的通断和权限"**
- 66 个已注册路由，751 条用例测的是：`GET /api/leads` → 200？✅
- 测不了：用真实手机号登录招生专员 → token 解析出来的 role 是什么 → 实际权限是什么
- 登录→token→查库→role 推断 这条链路的端到端验证，自动化完全没做

**3. 未注册的端点不会被自动化发现**
- `/api/workers/status-counts` 等缺失端点不在 66 个已注册路由中
- 自动化不会去测"不存在的路由"，只测存在的 66 个

**改进方向**：后续增加"真实账号登录 + 权限遍历"的集成测试流程。
