## 三、5 大类测试用例设计

### 第1类：正向功能测试

验证接口核心功能正常工作。
- 使用正确角色发起请求
- 提供完整合法的请求参数
- **预期**：HTTP 2xx + 返回标准数据结构 + 数据库记录实际写入

### 第2类：参数异常测试

验证接口对非法入参的防御能力。
- 缺失必填参数（phone、name、title 等）
- 参数类型错误（数字传字符串、数组传对象）
- 参数格式错误（手机号含字母、email 无@）
- **预期**：HTTP 400 + 明确错误提示

### 第3类：权限校验测试

验证接口的角色隔离和鉴权。
- 无 token 访问 → 401
- 伪造/过期 token → 401
- 客户尝试创建线索 → 403
- 讲师尝试审核简历 → 403
- 阿姨尝试查看他人简历详情 → 403
- **预期**：401 或 403 + 不暴露敏感数据

### 第4类：边界值测试

验证接口对极端数据的处理。
- 超长字符串（姓名 200 字符、内容 10000 字符）
- 空字符串 / null / undefined
- 负数（价格 -100、评分 -1）
- 超出范围值（评分 6、年龄 200）
- 特殊字符（SQL 注入 `' OR 1=1 --`、XSS `<script>alert(1)</script>`）
- **预期**：400 或正常处理（转义后入库）

### 第5类：重复操作测试

验证接口的幂等性和唯一约束。
- 同一手机号重复创建线索 → 409
- 同一手机号重复注册 → 409
- 连续2次快速创建相同订单 → 第二次应成功（不同ID）
- 同一人对同一目标重复评价 → 409 或覆盖
- 连续登录2次 → 各返回不同 token，均可用
- **预期**：409 冲突 / 正常创建不同记录

## 四、端到端业务流程测试

### 招生培训线（E01~E05h）

```
E01: 创建线索 → 跟进 → 发合同 → 主管确认签约
     └─ leads.status=signed + auto worker(pending) + auto student(signed)

E02: worker审核通过 → status=available

E03: worker报名课程 → student(training) + enrollment(enrolled)

E04: 讲师统一考核合格 → enrollment(passed) + student(qualified)

E05b: 转简历 → student(converted) + worker(available)

E05c: 考核未通过 → student(failed) → 重新培训

E05d: 主管建课 → admin审核通过 → 讲师排课 → 主管审课表
```

### 经纪人服务线（E06~E07e）

```
E06: 创建客户 → 创建订单 → 推荐阿姨 → 双方确认签约
     └─ order(signed) + customer(converted) + worker(working) + contract(active)

E07b: 换阿姨 → 旧签约replaced + 旧阿姨idle + 新阿姨working

E07e: 经纪人发起签约 → 阿姨手机端确认 → 双方确认生效
```

### 退款线（E08~E10）

```
E08: 招生发起培训费退款 → admin审核通过 → 确认完成
     refund: pending → approved → completed

E09: 经纪人发起中介费退款 → admin审核 → 完成

E10: 运营发起保证金退款 → admin审核 → 完成
```

## 六、测试文档输出规范

每条测试记录包含以下字段：

| 字段 | 说明 |
|------|------|
| **label** | 测试项名称，格式：`清单编号-简短描述` |
| **module** | 所属模块（auth/leads/workers/orders等） |
| **category** | 5大类之一：正向功能、参数异常、权限校验、边界值、重复操作 |
| **method** | HTTP方法（GET/POST/PUT/DELETE） |
| **url** | 请求路径 |
| **headers** | 请求头说明（Bearer token / 无鉴权） |
| **body** | 请求体参数（JSON格式） |
| **expect.status** | 预期HTTP状态码 |
| **expect.hasField** | 预期返回字段 |
| **失败判定条件** | 状态码不符 / 缺少关键字段 / 业务规则不符 |

## 九、接口问题定位方法

### 快速定位三步法

```
1. 看状态码
   ├─ 401 → token过期/未传，检查 Authorization header
   ├─ 403 → 角色权限不足，检查当前登录角色
   ├─ 400 → 参数错误，检查 body 字段名和类型
   ├─ 409 → 唯一约束冲突，手机号/ID已存在
   ├─ 404 → 路由不存在，检查 URL 路径
   └─ 500 → 服务端异常，查看终端日志

2. 看JSON报告
   ├─ reports/report_*.json → 筛选 status=FAIL
   ├─ 查看 error 字段的错误信息
   └─ 对比 expect 与实际返回值

3. 看服务端日志
   ├─ Next.js 终端输出
   ├─ Supabase/database 日志
   └─ 查看 SQL 是否实际执行
```
