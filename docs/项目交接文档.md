### 3.2 部署铁律

> **每次部署到 CloudBase 前，必须先运行 `pnpm validate`（tsc --noEmit + eslint）。**
> CloudBase 的 `next build` 遇到第一个类型错误就会失败，反复部署浪费积分（每次约 10 积分）。
> 不要依赖 `pnpm dev`（dev 模式不严格检查类型）。

### 3.3 数据层约定

- API 返回使用**下划线命名**（如 `resume_review_status`），前端映射函数需要兼容
- `src/lib/data-service.ts` 的 `mapXxxFromDb()` 函数优先读下划线 key，驼峰 key 作为 fallback
- 前端页面统一从 `@/lib/data-service` 导入数据和操作函数
- Mock 数据（`src/lib/mock-data.ts`）仅作本地开发回退

### 3.4 角色体系

- 8 种角色，customer 和 worker_operator 在注册入口隐藏但功能完整保留

## 5. 重要约定与配置

### 5.1 编码规范

- TypeScript strict 模式，禁止隐式 `any`
- 函数参数必须有类型标注
- 使用 `cn()` 合并 className（`src/lib/utils.ts`）
- 客户端组件标注 `'use client'`
- Mock 数据集中在 `src/lib/mock-data.ts`

### 5.2 环境变量

| 变量 | 说明 | 必配 |
|------|------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase 项目 URL | ✅ |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase 匿名公钥 | ✅ |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase 服务角色密钥 | ✅ |
| `WX_APPID` | 微信小程序 AppID | 生产必配 |
| `WX_APPSECRET` | 微信小程序 AppSecret | 生产必配 |
| `JWT_SECRET` | JWT 签名密钥 | 默认 `dev-secret-key` |
| `SMS_PROVIDER` | 短信服务商（如 `aliyun`） | 生产必配 |

### 5.5 数据库注意事项

- `users` 表有 `wechat_openid` 和 `wechat_unionid` 字段
- 测试用户 openid 格式：`dev_wx_{role}`（如 `dev_wx_agent`）
- 开发模式：未配置 WX_APPID 时，code 直接映射为 openid
- `orders.worker_id` 指向 `workers.id`（不是 `users.id`）
- 非 admin 角色查询 workers 只返回 `resume_review_status=approved` 的简历
