## 一、项目概览

| 项目 | 详情 |
|------|------|
| 平台定位 | 家政行业多角色 SaaS 协作平台 |
| 角色体系 | 8 种角色（阿姨/经纪人/招生代理/讲师/培训主管/阿姨运营/客户/管理员） |
| 技术栈 | Next.js 16 + React 19 + TypeScript 5 + Tailwind CSS 4 |
| 数据库 | Supabase PostgreSQL（15+ 业务表） |
| 部署 | CloudBase CloudRun 容器（Docker + pnpm） |
| 生产地址 | `https://szjfp-274552-8-1444411996.sh.run.tcloudbase.com` |
| 当前版本 | szjfp-015 |

### 核心需求

1. **8 种角色权限体系**：JWT 中间件（auth-middleware） + 侧边栏菜单白名单（DEFAULT_ROLES） + API 路由守卫（verifyAuth）三重保障
2. **简历审核流程**：阿姨简历修改后生成审核记录（带 diff 对比），管理员审核通过/拒绝
3. **线索 → 合同 → 订单全链路**：招生代理录入 → 跟进 → 签约 → 生成订单 → 阿姨接单
4. **佣金分账**：三级分账（经纪人 + 招生代理 + 平台），支持自定义比例
5. **诚信分 + 保证金 + 积分**：行为激励与风控闭环
6. **课程审批与排课**：讲师创建 → 培训主管审核 → 排课 → 学员报名 → 考核打分
7. **评价互评体系**：各角色可互相评价，按 target_role 分类
8. **双重代码目录**：`JZ-projects/src/` 和 `src/` 两套副本，修复需两边同步

## 三、关键架构决策

### 数据映射双路读取问题
早期 API 返回下划线 key（`resume_review_status`），但 `mapXxxFromDb` 用驼峰读取，导致所有字段返回默认值。第 7 轮根因修复后统一为下划线优先 + 驼峰 fallback，第 11 轮进一步清理死代码。

### 副本目录问题
项目同时存在 `JZ-projects/src/`（实际部署用）和 `src/`（副本）两套几乎相同的代码。每次修复需要两边同步，否则部署时生效的版本不对。

### 权限三层保障
1. **auth-middleware.ts**：全局 JWT 解析 + role 提取
2. **sidebar.tsx DEFAULT_ROLES**：前端菜单白名单（兜底）
3. **API 路由 verifyAuth**：接口级权限校验

### Sidebar 兜底优先级
DEFAULT_ROLES 作为菜单兜底 → API 返回的 page_access 作为补充 → 两者取并集。问题在于 DEFAULT_ROLES 最初只配了少数角色，导致非主流角色菜单缺失。
